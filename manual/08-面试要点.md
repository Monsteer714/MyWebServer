# 08 · HTTP 模块面试怎么说

> 这一章没有代码。只整理你在面试时应该主动讲的点，以及被追问时的回答策略。

---

## 开场：30 秒概述你的设计

> "我这个项目的 HTTP 模块做了分层设计。最底层是 I/O 层——epoll + 非阻塞 socket + Reactor/Proactor 双模。中间是协议层——HttpContext 解析器把字节流转成 HttpRequest 对象，HttpResponse 负责响应模型化。上层是路由和中间件层——Router 做路径分发，MiddlewareChain 用责任链模式处理横切关注点。每一层都可以独立测试，不依赖上层的 I/O。"

## 必被追问的话题及回答要点

### 追问1："为什么选择 token 分割而非正则来做动态路由？"

```
你有三层回答：

1. 性能：Token 分割 O(k)，k 是路径段数（通常 ≤ 5）；正则引擎是通用引擎，
   最坏 O(2^n)（灾难性回溯）。

2. 够用：RESTful API 的路径模式非常规整——/users/:id, /posts/:id/comments/:cid，
   token 分割能覆盖 95% 的场景。不需要正则为了一两个边缘 case 拖全局性能。

3. 可扩展：如果未来真的需要复杂匹配，可以用双表策略——先 Token 匹配快速路径，
   匹配不到再用正则兜底。这是分段优化的思路。
```

### 追问2："你的中间件链和 Express/Koa 的中间件有什么区别？"

```
关键点：before() 和 after() 的分离。

Express/Koa 的中间件用单个函数 + next() 调用链。你的是 before()/after()
显式分离，before 链在路由前正序执行，after 链在路由后逆序执行。

区别：
- Express 方式：中间件自己决定"在 next() 之前还是之后做事"，开发者可能忘掉 next()
- 你的方式：before/after 语义明确，不会出现"忘了调 next() 导致请求挂起"

反过来也要承认：Express 方式更灵活，可以在一个函数里同时做前置和后置逻辑。
你的方式更安全但表达能力稍弱——这是一个取舍。
```

### 追问3："解析器怎么处理粘包/半包？"

```
这是你设计中的核心亮点。

半包：HttpContext 是有状态解析器。如果一次 read 只拿到半个请求行，
parse() 返回 true（没有语法错），gotAll() 返回 false（没完成）。
上层继续 epoll_wait，下次数据到达后继续 parse()，状态机会从上次断点继续。

粘包：parse() 只读取 Content-Length 长度的 body。如果缓冲区里有多余数据，
reset() 后下次 parse() 会从剩余数据开始。
关键设计：数据缓冲区是 http_conn 持有的，HttpContext 不持有缓冲区，
只持有偏移量——所以"剩余数据"自然留在缓冲区里给下一个请求用。
```

### 追问4："HttpResponse 为什么模型化而不是直接拼字符串？"

```
三点：

1. 中间件兼容：CORS 中间件需要在 handler 执行之后给响应加 header。
   如果响应已经拼成字符串了，再插入 header 就要重新解析字符串——这是二次工作。
   模型化后直接 addHeader() 就行，最后的序列化 appendToBuffer() 统一处理。

2. 文件响应：我需要区分"发文本"和"发文件"。文本走 write()，文件走 sendfile()。
   模型化后用 isFileResponse() 标记，I/O 层据此选择不同的发送方式。
   字符串拼接做不到这种语义标记。

3. 错误处理：如果字符串拼接过程中一个 header 写错了格式（少了个 \r\n），
   调用方根本不知道——最终发出去的字节流是错的。模型化通过 appendToBuffer()
   统一格式，所有 header 都经过同一个出口，格式错误不可能发生。
```

### 追问5："HttpRequest 里 header key 的大小写问题怎么处理的？"

```
HTTP 协议规定 header 名称不区分大小写。Content-Type 和 content-type 等价。

我的做法：存储时统一转为小写。addHeader 时 tolower(key) 再存入 map。
查找时也 tolower 后查。

另一种做法是用自定义 comparator 传给 map——但统一小写更简单直观，
面试时容易讲清楚。
```

### 追问6："Keep-Alive 连接复用你的 reset() 怎么做的？"

```
核心是 swap 技巧。

http_conn::init() 重置 I/O 状态（缓冲索引、文件 fd、发送状态）。
HttpContext::reset() 重置解析状态（状态机回到 kExpectRequestLine）。
HttpContext::reset() 内部用 swap 把旧的 HttpRequest 和临时空对象交换——
这样旧数据的内存被回收，新对象是空的，避免了逐个字段清零的复杂度。

面试官问为什么用 swap：swap 是 O(1) 操作（交换内部指针），
如果要逐个字段清零（clear 每个 string、map），是 O(n) 且可能触发多次内存分配。
```

### 追问7："你的设计有什么不足？你怎么改进？"

```
不要回避这个问题。主动讲不足反而体现你反思过。

可以说：

1. 读缓冲区固定 2KB。如果 POST 请求 body 超大，解析器无法处理。
   → 改进：支持动态扩容，或者流式 body 处理（边收边处理，不全缓存）。

2. Router 的匹配是线性的。如果注册了 100 条路由，最坏 O(100) 次比较。
   → 改进：用 Trie 树索引路径前缀，匹配复杂度降到 O(路径深度)。

3. 中间件的 after() 在 handler 抛异常时会被跳过。
   → 改进：用 try-finally 包裹，确保 after 链一定执行。

4. 没有 HTTP/2 支持。
   → 说明：HTTP/2 是二进制协议，需要完全重写解析层。当前框架的协议层
       是面向文本的，要支持 HTTP/2 需要抽象协议层接口。
```

## "你参考了哪些设计"

这个问题面试官爱问，在考察你是否只是抄代码。

你应该这样答：

> "我参考了两个来源。一是 Kama-HTTPServer 的模块划分——它把 HttpRequest / HttpResponse / HttpContext 三个职责分得很清楚，我借鉴了这个分层思路。但 Kama 用的是 Muduo 网络库，我没有直接抄它的 I/O 层，而是把它的协议层设计适配到了我自己的 epoll 框架上。
> 
> 二是 Nginx 的 HTTP 解析器设计——它的状态机驱动方式影响了我。不过我简化了 Nginx 的复杂状态（Nginx 要处理太多边缘情况和反向代理场景），只保留了我需要的核心状态。"

这样说的好处：你承认参考了（诚实），但说清楚了参考了什么、自己做了什么，展示的是技术判断力和工程能力。

## 编码时对照面试

写完每个模块后，问自己三个问题：

1. **"面试官让我解释这个类的职责，我能不能 30 秒说清楚？"**
   → 说不清就说明类的职责不单一

2. **"这个函数为什么用这个签名而不是另一个？"**
   → 答不出就说明你没想清楚设计意图

3. **"这个模块能脱离其他模块独立测试吗？"**
   → 不能就说明耦合过重
